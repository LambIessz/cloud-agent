## 第四关：记忆系统（Memory Architecture）

```
我在学习 AI Agent 的记忆系统设计，该项目实现了「短期 + 长期」双层记忆架构：

【架构概览】
- 短期记忆（ShortTermMemory）：Redis，TTL=30分钟，存储当前会话对话历史
- 长期记忆（LongTermMemory）：Milvus 向量数据库，存储用户偏好（语义检索）
- 记忆提取器（PreferenceExtractor）：会话结束时用 LLM 从对话中提取偏好 → 写入 Milvus
- 统一入口（MemoryManager）：协调两层存储，提供 save_conversation / load_preferences / finalize_session 三个核心接口

【会话生命周期】
1. 用户首次提问 → load_preferences(user_id) 从 Milvus 检索历史偏好 → 注入 memory_context
2. 每轮对话 → save_conversation() 追加到 Redis
3. 会话结束 → finalize_session() 调用 LLM 提取新偏好 → 存 Milvus → 清空 Redis

请帮我深度理解：
1. 为什么短期用 Redis、长期用 Milvus？能不能都用 Redis 或者都用数据库？各自解决了什么核心问题？
2. load_preferences 的 top_k 默认是 3，为什么不检索全部偏好？向量数据库的语义检索在这里的作用是什么？
3. background_extract 和 finalize_session 都能提取偏好，两者的使用场景有什么区别？为什么 background_extract 不清空 Redis？
4. 如果用户说「我喜欢详细的技术解答」，这条偏好是如何被提取、向量化、存储、再被召回的？请追踪完整链路。
5. 当 Redis 或 Milvus 不可用时，代码如何做优雅降级？从代码层面找出降级逻辑。
```

## 第六关：deep_research 研究工作流

```
我在深入学习 deep_research 项目的多 Agent 研究工作流，其核心流程如下：

【工作流节点】
intent → (direct/multiagent) → plan → [web_search || local_rag] → deep_dive → analyze → (reflect/write)

【各节点职责】
- intent_node：规则引擎 + LLM 双层判断，决定「直接回答」还是「启动研究流程」
- plan_node：拆解研究问题 → 生成大纲 → 制定搜索计划（search_plan）
- web_search_node：执行 Bocha API 搜索 → 结果过滤 → LLM 整理结构化证据
- local_rag_node：向量检索本地知识库 → 结果过滤 → LLM 整理结构化证据
- deep_dive_node：证据评分 + 冲突审计 → 生成 evidence_pool
- analyze_node：基于证据池归纳结论 → 判断是否需要补充研究（needs_more_research）
- reflect_node：生成补搜计划（supplementary_queries）→ 触发新一轮 web+local 检索
- write_node：基于 findings 撰写 Markdown 研报 → 引用 ID 校验 + 修正

【关键设计】
- web_search 和 local_rag 是并行执行的（两条边都从 plan 出发指向 deep_dive）
- analyze → reflect → web_search 形成反思迭代循环，最多执行 max_iterations 次
- 每个节点只接收当前指令（不传 state["messages"]），避免 Token 累积

请帮我深度分析：
1. plan → web_search 和 plan → local_rag 两条边在 LangGraph 中如何实现真正的并行执行？StateGraph 的 fan-out/fan-in 机制是什么原理？
2. `needs_more_research` flag 控制是否触发反思迭代，这个 Reflection 机制解决了普通单次 RAG 的什么核心缺陷？
3. 证据评分函数 `_score_evidence` 使用了 domain 域名启发式规则来判断可信度，这种方法的优缺点是什么？有没有更好的方案？
4. write_node 中 `_validate_and_fix_citations` 校验并移除非法引用 ID，这个防幻觉机制解决了什么问题？不做这个校验会发生什么？
5. `with_memory_context` 函数在每个节点的 prompt 里注入跨会话记忆，这在研究场景下的典型使用场景是什么？
```
