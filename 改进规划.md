## 原始四项

### 1. 独立 eval 数据集 / golden set / trajectory 评分闭环

位置：`ops/`、`cloud_agent/agent/test/`、`cloud_agent/front/cloud_agent/src/`

现状：已有 smoke、release gate、browser smoke 和部分定向测试，但更多是在验证“能跑通”，不是稳定评价“是否更好”。

缺陷：缺少固定样本集、trajectory 级评分和统一指标。路由、工具调用、越权拦截、FinOps 幻觉、推广物料、故障排查和引用质量很难长期对比。

改进方案：先做一个小而稳的 golden set，把路由、工具选择、越权攻击、FinOps 幻觉、推广物料、故障排查和引用质量都变成可重复样本，再写入统一 eval 报告。

状态：已完成

### 2. 显式人机介入 checkpoint

位置：`cloud_agent/agent/core/workflow/graph_manager.py`、`cloud_agent/app/service/chat_service.py`、`cloud_agent/agent/core/workflow/state.py`

现状：现有流程大多是自动路由、自动执行、自动结束；只有发布或烟雾门禁里有人审，没有进入 Agent 执行链路的 checkpoint。

缺陷：低置信度决策、高风险写操作、外部调用失败重试等场景，缺少“先停一下等确认”的机制，也缺少从中断点恢复的显式状态。

改进方案：在高风险动作前插入确认节点，把 pending action、用户反馈和恢复上下文写入状态；确认后继续执行，拒绝后回退或换路由。

本轮改进：
- 在 `orchestrator` 里把高风险请求先路由到 `checkpoint_agent`，避免直接进入执行链路。
- 新增 `checkpoint_agent`，支持 pending / confirmed / rejected 三态，并在确认后回流到 `support_agent`。
- `chat_service` 把 pending checkpoint 持久化到短期记忆，恢复时优先加载并绕过语义缓存。
- `AgentState` 增加 `human_checkpoint`，图编排把 checkpoint 节点纳入状态机。

状态：已完成

### 3. 工具契约收敛

位置：`cloud_agent/agent/core/mcp/mcp_manager.py`、`cloud_agent/agent/mcp_servers/cloud_platform_server.py`、`cloud_agent/agent/agents/*.py`

现状：工具能用，但存在语义接近或重复的入口，工具返回格式也不是完全统一。

缺陷：LLM 容易被近义工具名带偏；调用方也要按工具逐个解析结果，异常、错误码、用户提示的表达不够一致。

改进方案：把工具输出统一成稳定结构，保留 `status/data/user_message/error_code` 字段；对重复或近似工具做合并、别名或下线，避免一个能力开多个口子。

状态：已完成

### 4. SupportAgent 从规则说明走向真实诊断闭环

位置：`cloud_agent/agent/agents/support_agent.py`、`cloud_agent/agent/core/workflow/graph_manager.py`

现状：SupportAgent 现在更像“按关键词生成排查清单”，依赖本地规则文件和静态建议，不是真正的诊断执行器。

缺陷：它能给方向，但无法验证实例状态、网络状态、监控指标或日志证据；用户补充信息不足时，容易停在泛化建议里。

改进方案：保留规则型首轮分诊，但补上只读诊断工具链，让它能逐步拉取实例、网络、监控、日志等证据，再把结果回填到下一轮推理。

本轮改进：
- SupportAgent 保留原有关键词分诊和排查清单，同时在识别到 `instance_id` 时进入只读诊断流程。
- 为 support agent 接入 MCP 工具 allowlist：`query_user_instances`、`analyze_instance_usage`。
- 诊断结果写入 `metadata.support_diagnostics`，包含状态、目标实例、工具列表、证据数量和摘要。
- 增加结构化事件：`support_diagnostic_tool`、`support_diagnostic`、`support_response`，方便后续观测层继续接入。
- `AgentGraphManager` 支持注入 `SupportAgentNode`，便于测试和后续替换真实诊断工具链。

## 新增六项

### 5. 任务规划层

位置：`cloud_agent/agent/agents/orchestrator.py`、`cloud_agent/agent/core/workflow/graph_manager.py`

现状：现在更像“先分类，再跳路由”，没有独立 planner，也没有把复杂请求拆成子任务、依赖和回退路径。

缺陷：一旦任务较长或较复合，系统容易靠一次路由硬跑到底；失败后多半直接 fallback，而不是重新规划。

改进方案：加一层轻量 planner，让它先产出子任务序列和停止条件，再交给执行层；当某一步失败时，允许重规划而不是只做单次分流。

本轮改进：
- 新增 `cloud_agent/agent/agents/planner_agent.py`，用确定性规则生成子任务序列、停止条件、followup agent 和重规划信息。
- `orchestrator` 对显式重规划和“先...再...”这类多步请求优先路由到 `planner_agent`。
- `AgentGraphManager` 接入 `planner_agent` 节点，并把 planner 的输出接到后续执行节点。
- `AgentState` 增加 `task_plan`，`context_manager` 补充 planner profile 和任务规划上下文注入。
- `task_plan` 事件与 planner 状态字段已经进入运行链路，后续可继续扩展失败重规划触发源。

### 6. 多 Agent 协作层

位置：`cloud_agent/agent/core/workflow/graph_manager.py`、`cloud_agent/agent/core/workflow/state.py`

现状：现在大多数路径是 `START -> orchestrator -> 单个 agent -> END`，真正的协作较少，主要只有 billing 到 finops 的有限接力。

缺陷：Agent 之间更像并行孤岛，缺少共享中间结果、合并结论和交叉校验的能力，组合型任务需要重来一遍。

改进方案：定义更清晰的 handoff state，让多个 agent 可以按顺序协作；必要时加一个 synthesis 节点来合并结论、消解冲突。

本轮改进：
- 新增 `cloud_agent/agent/core/workflow/collaboration_state.py`，把协作模式、参与者、finding、交叉校验和合并结论显式写入状态。
- `orchestrator` 在 FinOps workflow 中初始化协作状态，确保 billing -> finops -> synthesis 这条链可被图编排识别。
- `billing_agent` 和 `finops_agent` 会把各自输出追加为可合并的 finding，保留中间证据。
- `AgentGraphManager` 新增 `collaboration_agent` 节点，并让 `finops_agent` 在协作状态存在时转入 synthesis。
- 新增 `CollaborationSynthesisAgent`，合并账单侧和成本侧结论，并做资源 ID 交叉校验。

### 7. 上下文管理层

位置：`cloud_agent/agent/core/workflow/state.py`、`cloud_agent/app/service/chat_service.py`、`cloud_agent/agent/core/memory/`

现状：现在有 `messages` 和 `memory_context`，也有短期 / 长期记忆，但还没有明确的 token 预算、摘要压缩和分角色上下文裁剪。

缺陷：长对话越跑越大时，关键约束、工具结果和历史决策容易混在一起；系统不够清楚该保留什么、压缩什么、丢弃什么。

改进方案：做一个统一的上下文管理策略，把近期原文、摘要记忆、工具结果和系统约束分层保存，再按预算拼装给不同 Agent。

本轮改进：
- 新增 `cloud_agent/agent/core/workflow/context_manager.py`，把系统约束、历史摘要、近期原文、长期偏好和工具/诊断备注分层组装成 `context_bundle`。
- `MemoryManager` 新增 `build_context_bundle`，`chat_service` 在入图前注入 bundle，并给各 Agent 提供按角色裁剪的上下文。
- `orchestrator`、`product_agent`、`billing_agent`、`promotion_agent`、`recommendation_agent` 改为通过 `select_agent_memory_context` 读取角色上下文。
- `AgentState` 增加 `context_bundle`，保留 `memory_context` 作为默认兼容入口。

### 8. SSE / 流协议层

位置：`cloud_agent/app/service/chat_service.py`、`cloud_agent/app/router/chat.py`、`cloud_agent/front/cloud_agent/src/utils/`

现状：现在已经有 `stream_start`、`route_decision`、`tool_call_start`、`tool_call_end`、`message_delta`、`final`、`done` 这些事件，前后端也都能消费。

缺陷：协议虽然能用，但版本化和契约感还不够强；后续一旦扩事件类型或改字段，很容易让前端和后端悄悄漂移。

改进方案：给 SSE envelope 定一个稳定 schema 和版本号，把前后端解析逻辑和事件顺序写成契约测试，未知事件走兼容分支。

本轮改进：
- 后端 SSE payload 统一写入 `schema_version: "1.0"`。
- `/api/chat` 响应头增加 `X-SSE-Schema-Version: 1.0`。
- 前端请求显式声明 `Accept: text/event-stream`，状态归约器记录 `schemaVersion`，未知事件按兼容分支忽略。
- smoke / 后端 / 前端测试同步校验 schema 版本、响应头和事件契约。

### 9. 工具发现 / 路由规模化

位置：`cloud_agent/agent/core/mcp/mcp_manager.py`

现状：现在主要还是静态 allowlist，工具少的时候很清楚；但能力一多，就会越来越依赖名字和人工维护。

缺陷：缺少按能力检索、按场景筛选、按相似度收敛的机制。工具量上来后，LLM 看到的候选集会越来越杂。

改进方案：给工具补能力标签、用途标签和约束标签，保留 allowlist 作为安全边界，再加一层轻量检索或打分，把候选工具先收窄再交给模型。

本轮改进：
- `mcp_manager.py` 增加工具发现目录和 agent 发现画像，按 query 对 allowlist 内工具做轻量打分和收敛。
- `get_tools_for_agent()` / `get_tool_names_for_agent()` 支持传入 query，返回更小的候选集，并发出 `mcp_tool_selection` 事件。
- `billing_agent`、`finops_agent`、`promotion_agent`、`recommendation_agent`、`support_agent` 改为把最后一条用户消息传给工具发现层。
- 账单、支持、推广、推荐这几类场景的工具排序已按 query 验证，避免无关工具混入候选集。

### 10. 观测层

位置：`cloud_agent/agent/core/workflow/event_log.py`、`cloud_agent/agent/core/workflow/tool_audit.py`、`cloud_agent/agent/core/workflow/degradation_audit.py`

现状：现在已经有 EventLog、ToolAudit 和 Degradation 事件，但运行时仍能看到散落的 `print`，日志形态也还不算完全统一。

缺陷：人工排障能看懂，但机器采集、长期分析和统一检索的体验还不够顺手；细节信息也容易散落在不同输出里。

改进方案：继续把运行时输出收敛成结构化日志，统一字段和脱敏规则，把调试信息和正式观测信息分开，减少噪声。

本轮改进：
- 新增统一的结构化观测输出层，`EventLog` / `ToolAudit` / `Degradation` 都通过同一套 logger 发送 JSON。
- 增加观测脱敏规则，默认遮蔽 `query`、`prompt`、`completion`、`message`、`user_id`、`session_id`、`authorization`、`api_key` 等字段。
- 对嵌套结构也做递归脱敏，避免把原文上下文或 API key 前缀带进日志。
- 把路由器里散落的 `print` 调试输出改成 `logger.debug`，减少运行时噪声。
