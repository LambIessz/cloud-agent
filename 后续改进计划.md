## 一、关键约束（始终不可破坏）

继续工作时必须保持：

- `[ToolAudit]` / `[Degradation]` / `[Orchestrator]` 原始 print 输出
- request_id 生成、贯穿 EventLog / 日志 / SSE done 返回

继续工作时禁止：

- 把 OpenTelemetry Trace 扩展到 Web `stream_chat` request span 之外
- 引入 `prometheus-client` 依赖
- 跳过既有回归集
- 在 EventLog 或 metric label 中输出明文 user_id、prompt、completion、query、matched_question、对话内容、偏好内容、异常 message

### 2.5 副线 C — SemanticCache `load_collection` 缺失

完成时间：2026-05-25

发现：步骤 1.3 在 SemanticCache 可用之后，第二次发同一 query 仍报 `MilvusException: code=101, Collection 'qa_semantic_cache_v2' is in state 'released', call load() before search/get/query`。

根因：[cache.py:255-287](cloud_agent/app/infra/cache.py#L255-L287) 的 `_ensure_collection` 仅在 collection 不存在时创建。preload 写完数据进程退出后 milvus-lite 把 collection 落盘为 `released` 状态。backend 重启时 `has_collection` 返回 true，跳过创建，但从来没有 `load_collection`，导致任何 search 都失败。

修复：[cache.py:48-49](cloud_agent/app/infra/cache.py#L48-L49) 在 `_ensure_collection()` 之后加 1 行：

```python
self._ensure_collection()
self._client.load_collection(COLLECTION_NAME)     # 新增
self._available = True
```

风险评估：

- 不改 set_cache / get_cache 任何返回值或副作用
- 不改 EventLog、metrics、ToolAudit、Degradation 任何字段
- 不改 milvus-lite db 文件格式或 schema
- 只在 `initialize()` 内多一次 collection load 调用（毫秒级）

### 5.1 semantic_cache `set_cache` 在运行时写入

状态：已完成（2026-05-25）。

本轮已把 `stream_chat` 的 cache miss 路径接入运行时写入：当 Agent 工作流返回最终回答后，后台异步调用 `semantic_cache.set_cache()` 写入 user-scoped cache。写入内容包含 query / response；如果最终 `AIMessage` 带 LangChain `usage_metadata` 或 `response_metadata.token_usage`，同时写入 `estimated_prompt_tokens`、`estimated_completion_tokens`、`estimated_cost_usd` 和 `model`。

- `set_cache()` 新增 `raise_on_error` 可选参数；运行时写入使用 `raise_on_error=True` 交给 `chat_service` 统一降级。
