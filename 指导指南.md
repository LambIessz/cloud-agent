## 分析维度二：架构层缺陷分析
深入阅读源码，逐一检查以下 7 个维度。**每个维度必须定位到具体文件和函数，禁止泛泛而谈。**
---
### 1. 任务规划（Task Planning）
分析项目当前使用的规划范式（ReAct / Plan-and-Execute / CoT / 无显式规划）。
重点检查：- 任务能否被分解为子任务？有无显式的 Plan 生成步骤？- 子任务执行失败时，是否有动态重规划机制，还是直接报错退出？- 规划结果是否可被持久化，以支持断点续跑？- 若项目完全没有规划层，直接依赖 LLM 单步决策，是否存在引入规划层的空间？
---
### 2. 多 Agent 协作
分析项目的 Agent 拓扑结构。
重点检查：- 当前是单 Agent 还是多 Agent 架构？- 若是单 Agent：是否存在天然适合拆分为 Orchestrator + Worker 结构的场景？- 若已是多 Agent：通信协议是什么？角色职责是否清晰？结果聚合策略是否合理？- Agent 之间是否存在状态共享机制，还是完全无状态地串行调用？
---
### 3. 上下文管理策略（Context Management）
分析项目如何处理长对话和 token 限制问题。
重点检查：- 当 context 接近模型 token 上限时，策略是什么（硬截断 / 滚动窗口 / 摘要压缩）？- 压缩或截断策略是否会丢失关键的工具调用历史或中间结果？- 跨轮次的对话历史如何存储和读取？是否存在冗余或浪费？- 是否有针对不同类型信息（系统指令 / 工具结果 / 对话历史）的差异化管理策略？
---
### 4. Human-in-the-loop 机制
分析 Agent 在执行过程中是否支持人工干预。
重点检查：- 遇到低置信度决策或高风险操作（文件写入、外部 API 调用、消息发送等）时，  是否有暂停并等待人工确认的机制？- 是否有 Checkpoint 机制，支持 Agent 执行中断后从断点恢复？- 人工介入后，Agent 能否将人的反馈融入后续决策，还是只能重头开始？
---
### 5. Agent 评估框架（Evaluation）
分析项目是否具备系统评估 Agent 表现的能力。
重点检查：- 是否有 eval 模块或测试集？评估粒度是只看最终结果，还是追踪完整 trajectory？- 是否集成了外部评估工具（LangSmith / Arize / 自定义 pipeline）？- 工具调用的选择是否正确、参数是否合理，有没有被量化追踪？- 若完全没有评估机制，引入最小可用 eval pipeline 的改造成本是多少？
---
### 6. Tool 检索与路由（Tool Retrieval）
分析项目的工具管理和选择机制。
重点检查：- 当前是全量工具注入（所有 tool schema 塞入 system prompt）还是按需检索？- 若工具数量超过 20 个，是否有向量化检索机制按任务动态选取相关子集？- 工具的描述质量如何？是否存在语义相似导致 LLM 选择混乱的问题？- 是否支持 MCP 协议？若不支持，现有 tool 层是否有接入 MCP 的改造空间？
---
### 7. Streaming 与中间状态可见性
分析 Agent 执行过程的实时可见性。
重点检查：- 是否支持流式输出（streaming token by token）？- Agent 的中间状态（当前在执行哪个子任务、调用了什么工具、参数是什么）是否对外暴露？- 是否有事件/状态订阅接口，供前端实时展示 Agent 执行进度？- 若缺失，现有架构是否支持以低侵入性方式引入 streaming 和状态广播？

## 实现原则

**逐步实现，每步可验证**
严格按照方案中的实现步骤顺序执行，每完成一步，
项目必须仍处于可运行状态，不允许出现中间状态的残缺代码。

**最小侵入性**
- 优先扩展，而不是修改现有函数
- 若必须修改现有函数，只改必须改的部分
- 新增的公共接口保持向后兼容

**风格一致性**
新代码的命名、注释、类型标注风格必须与现有代码保持一致，
不要引入项目中没有使用的新依赖，除非确实必要。

## 实现流程

### Step 1：接口定义（先定义，后实现）

首先只定义新增的核心接口、类签名、函数签名，
不写具体实现（函数体用 `pass` 或 `raise NotImplementedError`）。

完成后停下来，输出：
- 新增了哪些文件 / 接口
- 这些接口如何与现有代码衔接
- 有无发现方案设计时遗漏的依赖或冲突

确认接口设计合理后，再进入下一步。

### Step 2：核心逻辑实现

按照实现步骤依次完成每个文件的具体逻辑。

每个文件完成后输出：
- 文件路径
- 实现了什么
- 与方案设计有无偏差，若有，说明原因

### Step 3：集成到现有代码

将新模块接入现有的调用链。

重点检查：
- 原有的调用方式是否仍然有效（不破坏现有用法）
- 新功能的触发路径是否完整
- 有无遗漏的错误处理分支

### Step 4：验证

按照方案中的验证策略逐项执行：

**手动验证：**
给出一个最小可运行的 demo 脚本，
直接在终端可以跑通，证明核心功能有效。

**单元测试：**
为以下场景各写一个测试用例：
- 正常路径（happy path）
- 边界情况（如空输入、超长输入、工具调用失败）
- 与现有功能的兼容性（原有用法不受影响）

测试文件放在项目现有的测试目录下，命名风格与现有测试文件一致。
