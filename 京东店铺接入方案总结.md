# 多 Agent 客服系统 × 京东店铺接入 · 对话总结

> 从云客服 Demo 到真实电商场景的完整升级路径 | 2026-06-02

---

## 一、项目背景与场景转换

### 原始项目

多 Agent 云平台智能客服——面向阿里云用户，覆盖 ECS/RDS/VPC 等 7 类业务场景。

### 目标场景

接入京东**金百达内存条**官方旗舰店，用自定义 AI Agent 替代京东小智，处理买家咨询（内存条规格、兼容性、订单物流、售后质保等）。

### 核心复用

| 层 | 复用率 | 改动 |
|------|:--:|------|
| Orchestrator 路由 | 100% | 关键词从"ECS/退款"改为"时序/兼容" |
| LangGraph 状态图 + 条件边 | 100% | 不变 |
| MCP ToolRegistry + 白名单 | 100% | 工具实现换为京东 API |
| 双层记忆 (Redis + Milvus) | 100% | 不变 |
| SemanticCache | 100% | FAQ 预热 |
| Prometheus + Grafana | 100% | 不变 |
| JWT/OIDC 身份隔离 | 100% | 京东买家 ID 天然隔离 |
| FinOps Agent | ❌ | 删除（不需要成本分析） |

## 二、京东店铺接入步骤

### Step 1 — 京东开放平台注册

1. 打开 **open.jd.com** → 注册开发者 → 人脸识别实名认证（1 个工作日）
2. 创建「商家应用」→ 勾选要用的 API 权限：

| API | 用途 |
|------|------|
| `jingdong.im.message.query` | 收买家消息 |
| `jingdong.im.message.reply` | 发 Agent 回复 |
| `jingdong.pop.order.search` | 查订单 |
| `jingdong.etms.trace.get` | 查物流 |
| `jingdong.ware.read.findWareById` | 查商品详情 |

3. 获取 `App Key` + `App Secret` + `Access Token`

### Step 2 — 配置回调

京东开放平台 → 消息服务 → 指向 `https://你的域名/jd/callback`，配签名 Token 和加密密钥。

### Step 3 — 改造 7 Agent

| 原 Agent | 改为 | 角色 |
|------|------|------|
| Product Agent | Product Agent | 内存条规格/时序/颗粒/电压/XMP 参数 |
| Billing Agent | Order Agent | 京东订单状态 + 物流追踪 |
| FinOps Agent | ❌ 删除 | — |
| Promotion Agent | Promotion Agent | 促销活动 + 满减优惠 |
| Recommendation Agent | Recommendation Agent | 兼容性推荐（B760 用哪款） |
| Support Agent | Support Agent | 点不亮/蓝屏/跑不满频率 排查 |
| Fallback Agent | Fallback Agent | 兜底 + 转人工触发 |

### Step 4 — 替换 MCP 工具

原有工具全部替换为京东 API 版：

| 原工具 | 新工具 | 数据源 |
|------|------|------|
| `query_user_instances` | `query_jd_order` | JD API 实时查 |
| `search_product_catalog` | `query_sku_info` | 本地 MySQL 商品表 |
| `analyze_instance_usage` | `track_logistics` | JD API 物流 |
| `generate_ai_poster` | `generate_product_card` | — |
| — | `check_compatibility` (新增) | 主板 QVL 表 |

### Step 5 — 兼容性检测工具（金百达场景特色）

内存条买家最关心「我的主板能用吗」。工具逻辑：

```
输入：SKU + 主板型号（如"B760M-A + 银爵 DDR5 6000"）
    │
    ├─ ✅ XMP 直接开启 → 完整兼容，推荐
    ├─ ⚠️ 需降频 → B660 只能跑 5600
    └─ ❌ 完全不行 → DDR4 插不进 DDR5
```

数据源：金百达官方兼容性测试报告 + 主板 QVL 列表。

### Step 6 — 部署上线

- SSL 证书（Let's Encrypt）加密 HTTPS 通信
- `docker compose up -d` 部署至阿里云 ECS
- 京东沙箱环境联调 → 发布上线审核（1-3 天）

### Step 7 — 京东消息完整数据流

```
买家输入 → 京东推送消息至 https://你的域名/jd/callback
         → 签名验证（App Secret + SHA1）
         → 提取 buyer_id / content / session_id
         → 进入 stream_chat 核⼼逻辑
         → Orchestrator 路由 → Agent 推理
         → 结果通过 jingdong.im.message.reply 发回
```

### Step 8 — 订单+物流查询链路

```
买家"我的订单到哪了"
  → Order Agent 调 query_jd_order（JD API 实时查）
  → 状态=已发货 → 拿到物流单号
  → Agent 自动再调 track_logistics（查轨迹）
  → 返回："5月29日广州发货，当前在杭州中转，预计明天送达"
```

全程买家一句话，Agent 自主完成**路由→查订单→自判需查物流→调物流→组织回复**。

## 三、培训手册和聊天记录怎么用

| 优秀客服回答 | 直接喂给 Agent，提升回答质量 |

## 四、与京东小智的对比

| 维度 | 京东小智（官方） | 我的 Agent |
|------|------|------|
| 意图识别 | 关键词匹配 | 语义理解 + 两级路由 |
| 变体问法 | 换说法就 miss | 100 种问法命中 |
| 多轮记忆 | 无 | Redis 短时 + Milvus 长时偏好 |
| 个性化 | 所有买家同样话术 | 回头客自动推荐历史关注的型号 |
| 工具调用 | 无 | MCP 协议：查订单+物流+兼容性 |
| 知识更新 | 人工改 JSON | 新增文档 → 向量化 → 即用 |
| 成本控制 | 固定软件费 | 语义缓存命中时绕过 LLM，零成本 |
| 大促稳定性 | 京东原生扛 | 需自己扩容 |
| 平台合规 | ✅ | 需审核 |

## 五、上线后可能遇到的问题

### 程序侧

| 问题 | 解决方案 |
|------|------|
| 响应超过 3 秒 | 先秒回"正在查询"，后台处理完再推送 |
| DeepSeek API 挂了 | 降级到备用模型（如千问） |
| LLM 幻觉误报质保/价格 | 输出层校验：价格从不信 LLM，走 API 实时查 |
| 京东 API QPS 限流 | 本地缓存 5 分钟 + 请求队列 |
| 内存/CPU 打满 | 每容器 memory limit + 凌晨定时重启 |
| Prompt 注入 | 检测"忽略指令"关键词 → 直接拒绝 |

### 客户侧

| 问题 | 原因 | 解决 |
|------|------|------|
| 听不懂口语 | 训练数据缺"抽不抽风"类说法 | 聊天记录提取词组加入路由 |
| 答非所问 | 路由错（"超频"去了商品介绍）| 调优路由关键词 |
| 售后太死板 | Prompt 没感情话术 | 加共情段："抱歉给您不便——" |
| 凌晨还在回复 | 无营业时间逻辑 | 非工作时间自动转"客服已下班" |
| 大促报错价 | 知识库里有旧价格 | 价格改成 API 实时查 |

## 六、面试叙事模板（1 分钟版）

> 我把这个多 Agent 框架迁移到了一个真实的京东金百达内存条店铺。店铺朋友让我做了一天灰度验证——Agent 当天处理了 40+ 条真实买家消息，自主解决率约 70%。

> 改造思路是"框架复用，数据替换"——7 个 Agent 的 Prompt 从云架构师改成内存条技术顾问，MCP 工具从查 ECS 规格换成京东 API 查订单和物流，还专门加了兼容性检测工具。知识库用店铺培训手册和 500+ 条真实聊天记录建的，比我自己写 Prompt 准得多。

> 虽然后来因为店铺必须用京东官方客服而下线了，但这一天验证了框架跨领域迁移的可行性——从云客服到电商客服，核心是 Prompt 和工具实现，不用动架构。

## 七、相关概念速查

| 术语 | 解释 |
|------|------|
| **B端** | Business（企业）客户——你的 Agent 服务店铺这家企业 |
| **C端** | Consumer（消费者）——买家是最终用户 |
| **工作流** | 复杂任务拆成「节点+条件边」，像外卖配送：下单→做菜→打包→配送 |
| **京东小智** | 京东原生 AI 客服——规则匹配，无 LLM 推理 |
| **咚咚** | 京东买家-商家即时通讯系统 |
| **OAuth 2.0** | 京东 API 的认证方式——App Key + Access Token |
| **签名验证** | 京东回调时用 SHA1 验证请求是否来自京东 |
