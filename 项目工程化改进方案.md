# 企业级 AI Agent 项目工程化改进方案

## 进行项目改造时必看
进行改造任务时，需要考虑后续的任务，所以改造时的代码需要谨慎

## 1. 文档目标

本文档用于指导后续对 `cloud_agent` 和 `deep_research` 两个子系统进行工程化改造。目标不是简单增加功能，而是把前面学习中发现的问题整理成可执行的开发任务，逐步提升系统的可扩展性、安全性、稳定性、可观测性和面试表达质量。

项目当前已经具备多 Agent 编排、RAG、语义缓存、短期记忆、MCP 工具调用和流式输出能力，但仍存在一些典型 Demo 到生产之间的差距：

- Orchestrator 路由缺少显式 fallback 和多意图处理。
- `cloud_agent` 缺少故障排查类 Agent。
- MCP Client 每次请求重复创建，工具发现和连接生命周期不够清晰。
- 用户身份主要来自请求参数，生产环境需要接入后端认证上下文。
- Prompt 约束已有，但关键业务事实缺少程序化校验。
- `deep_research` 的反思迭代主要依赖 LLM 输出，缺少额外防护条件。
- 当前日志偏开发调试，缺少结构化日志、指标、trace 和成本监控。
- 短期记忆已有裁剪，但中途偏好提取机制还没有接入 Web 链路。

## 2. 改造优先级

### P0：先做，直接影响项目完整性和面试可信度

- 新增 `support_agent`，覆盖故障排查场景。
- 增加 `fallback_agent` 或 `out_of_scope` 路由兜底。
- 优化 Orchestrator 路由规则，处理复合意图。
- 增加 Redis / Milvus / MCP 工具调用的降级日志。
- 增加核心链路的 request_id。

### P1：中期做，提升工程化质量

- MCP Client 生命周期统一管理，避免每次请求重复创建。
- 为工具调用增加超时、重试、失败统计和审计日志。
- 给 FinOps 输出增加事实校验，避免编造实例 ID、监控指标和节省金额。
- 给 `deep_research` 的反思循环增加防护条件。
- 接入中途偏好提取，每 N 轮自动调用 `background_extract`。

### P2：后续增强，接近生产级能力

- 接入认证上下文，避免前端直接传可信 `user_id`。
- 引入结构化 JSON 日志。
- 增加 Prometheus Metrics 和 Grafana Dashboard。
- 接入 OpenTelemetry Trace。
- 增加缓存命中率、Token 成本、Agent 节点耗时等监控。
- 将 stdio MCP Server 改造为长生命周期服务或远程服务。

## 3. 改造任务一：新增 support_agent

### 目标

当前 `cloud_agent` 覆盖产品咨询、账单查询、推荐、推广、FinOps 成本优化，但缺少技术支持和故障排查场景。新增 `support_agent` 后，系统可以处理如下问题：

- ECS 无法 SSH 连接。
- 安全组端口不通。
- 公网 IP 无法访问。
- CPU 或内存异常升高。
- RDS 连接失败。
- 实例启动失败或状态异常。

### 涉及文件

- `cloud_agent/agent/agents/support_agent.py`
- `cloud_agent/agent/agents/__init__.py`
- `cloud_agent/agent/core/workflow/graph_manager.py`
- `cloud_agent/agent/agents/orchestrator.py`
- 可选：`cloud_agent/mock_data/ecs_troubleshooting_guide.md`

### 实现思路

1. 新建 `SupportAgentNode`。
2. Prompt 中明确它只负责故障排查，不处理账单、推荐、推广问题。
3. 初版可以先基于已有 mock troubleshooting 文档和通用规则回答。
4. 后续可以接入 MCP 工具，例如查询实例状态、安全组规则、网络 ACL、监控指标。
5. 在 `AgentGraphManager` 中注册节点。
6. 在 Orchestrator 路由规则中新增 `support_agent`。
7. 增加 `support_agent -> END` 边。

### 验收标准

- 用户问“我的 ECS 无法 SSH 连接”时路由到 `support_agent`。
- 用户问“安全组 22 端口不通怎么办”时路由到 `support_agent`。
- `support_agent` 能给出分步骤排查清单。
- 不影响原有 product / billing / finops 路由。

## 4. 改造任务二：增加 fallback_agent / out_of_scope 兜底

### 目标

当前 Orchestrator 无法判断时默认路由到 `product_agent`。这会导致无关问题被硬塞给产品 Agent，例如：

- 今天天气怎么样？
- 帮我写一首诗。
- 给我推荐一家餐厅。

应该显式增加 fallback 机制，让系统对能力范围外的问题给出边界清晰的回复。

### 涉及文件

- `cloud_agent/agent/agents/fallback_agent.py`
- `cloud_agent/agent/core/workflow/graph_manager.py`
- `cloud_agent/agent/agents/orchestrator.py`

### 实现思路

1. 新增 `FallbackAgentNode`。
2. Orchestrator 允许输出 `fallback_agent` 或 `out_of_scope`。
3. Prompt 中增加能力边界说明。
4. 对非云平台相关问题，返回 fallback。
5. Fallback 回复内容包含：
   - 当前系统主要支持哪些问题。
   - 用户可以如何重述问题。
   - 不编造无关回答。

### 验收标准

- “今天天气怎么样”不会进入 `product_agent`。
- fallback 回复明确系统能力边界。
- 原有业务问题不被错误兜底。

## 5. 改造任务三：优化 Orchestrator 路由

### 目标

当前 Orchestrator 是单标签路由，遇到复合意图容易误判。例如：

```text
帮我推荐一个便宜的机器，我的账单太高了。
```

这个问题同时包含推荐和降本，但核心目标是降低成本，应优先进入 FinOps 流程。

### 涉及文件

- `cloud_agent/agent/agents/orchestrator.py`
- `cloud_agent/agent/core/workflow/state.py`
- `cloud_agent/agent/core/workflow/graph_manager.py`

### 实现思路

1. 增加硬规则优先级：
   - 同时出现“推荐/选型”和“账单高/降本/优化成本”，优先 `finops_agent_trigger`。
   - 同时出现“查资源”和“优化成本”，优先 `finops_agent_trigger`。
   - 只问“适合业务的配置”，走 `recommendation_agent`。
2. 引入两阶段路由：
   - 第一阶段：`in_scope / out_of_scope / need_clarification`。
   - 第二阶段：`product / billing / promotion / recommendation / finops / support`。
3. 引入多意图识别：
   - 主意图写入 `next_agent`。
   - 次意图写入 `metadata["secondary_intent"]`。
   - 路由理由写入 `metadata["route_reason"]`。
4. 保留默认兜底，但默认值从 `product_agent` 改为 `fallback_agent`。

### 验收标准

- 复合意图能稳定进入主目标 Agent。
- out-of-scope 问题进入 fallback。
- 路由结果、主意图、次意图、理由可在日志中查看。
- 路由准确率测试集保持或超过 90%。

## 6. 改造任务四：MCP Client 生命周期管理

### 目标

当前 BillingAgent 和 FinOpsAgent 每次调用都会重新创建 `MultiServerMCPClient` 并执行 `get_tools()`。低并发 Demo 可以接受，但高并发下会造成重复初始化、工具发现、stdio 子进程资源积累和延迟增加。

### 涉及文件

- `cloud_agent/agent/core/mcp/mcp_manager.py`
- `cloud_agent/agent/agents/billing_agent.py`
- `cloud_agent/agent/agents/finops_agent.py`
- `cloud_agent/app/app_main.py`
- `cloud_agent/app/service/chat_service.py`

### 实现思路

1. 在 `mcp_manager.py` 中实现 `MCPToolRegistry`。
2. 应用启动时初始化一次 `MultiServerMCPClient`。
3. 调用 `get_tools()` 后缓存所有工具。
4. 按 Agent 名称缓存白名单工具：
   - Billing：`query_user_orders`, `query_user_instances`
   - FinOps：`query_user_instances`, `analyze_instance_usage`
   - Promotion：`get_promotable_products`, `search_product_catalog`, `get_promotion_materials`, `generate_ai_poster`
5. Agent 不再直接创建 MCP Client，而是从 registry 获取工具。
6. 应用关闭时释放 MCP Client 资源。

### 验收标准

- 单次请求不再重复执行工具发现。
- BillingAgent 和 FinOpsAgent 不直接创建 `MultiServerMCPClient`。
- 工具白名单仍然生效。
- 多轮请求 MCP 初始化次数为 1。
- 有 MCP 初始化失败的降级日志。

## 7. 改造任务五：工具调用超时、重试和审计

### 目标

工具调用是 Agent 系统中最容易拖慢和出错的环节。需要增加超时、重试、错误统计和审计。

### 涉及文件

- `cloud_agent/agent/mcp_servers/cloud_platform_server.py`
- `cloud_agent/agent/core/mcp/mcp_manager.py`
- `cloud_agent/agent/agents/billing_agent.py`
- `cloud_agent/agent/agents/finops_agent.py`

### 实现思路

1. 为每次工具调用增加 timeout。
2. 对可重试错误进行有限重试，例如数据库临时连接失败。
3. 连续失败后熔断一段时间，避免拖垮系统。
4. 记录审计日志：
   - `request_id`
   - `user_id_hash`
   - `tool_name`
   - `latency_ms`
   - `status`
   - `error_type`
5. 工具失败时，Agent 给用户业务友好的回复，不暴露内部错误。

### 验收标准

- 工具超时不会导致请求无限挂起。
- 工具失败日志可追踪到具体 tool。
- 用户不会看到数据库连接串、堆栈或内部异常。

## 8. 改造任务六：FinOps 事实校验

### 目标

FinOps Prompt 已要求模型不要编造实例 ID、监控指标和节省金额，但 Prompt 是软约束。需要增加程序化校验。

### 涉及文件

- `cloud_agent/agent/agents/finops_agent.py`
- 可新增：`cloud_agent/agent/core/workflow/validators.py`

### 实现思路

1. 捕获工具返回结果中的合法实例 ID。
2. 捕获 CPU、内存、带宽等监控指标。
3. 在最终回答前校验：
   - 回答中的实例 ID 是否来自工具结果。
   - 回答中的监控数值是否来自工具结果。
   - 节省金额是否有明确计算依据。
4. 如果校验失败：
   - 移除不可信结论。
   - 或让模型基于工具结果重新生成。

### 验收标准

- 工具未返回实例 ID 时，回答不应编造实例 ID。
- 工具未返回监控数据时，回答不应给出具体 CPU / 内存数值。
- 节省金额没有计算依据时，只能给出定性建议。

## 9. 改造任务七：中途偏好提取

### 目标

当前 `finalize_session` 在会话结束时提取偏好。如果用户中途断开连接，本次会话中的偏好可能无法进入长期记忆。应基于已有 `background_extract` 增加每 N 轮自动提取机制。

### 涉及文件

- `cloud_agent/agent/core/memory/memory_manager.py`
- `cloud_agent/app/service/chat_service.py`

### 实现思路

1. 在 Redis 或内存中记录每个 session 的 turn_count。
2. 每完成一轮 user/assistant 对话后，turn_count + 1。
3. 当 turn_count % N == 0 时，异步触发 `background_extract`。
4. `background_extract` 只写 Milvus，不清空 Redis。
5. 抽取成功后记录日志并更新偏好缓存。

### 推荐参数

- N = 5，适合长对话中途存档。
- 最少 2 轮对话后再触发，避免噪声过大。

### 验收标准

- 连续对话 5 轮后自动触发一次偏好提取。
- Redis 历史不被清空。
- Milvus 中能看到新增偏好。
- 用户突然断开后，前面已触发的偏好不会丢失。

## 10. 改造任务八：deep_research 检索质量兜底

### 目标

`web_search_node` 当前每个查询固定返回若干结果。如果所有结果相关性都很低，系统可能仍然进入后续分析，导致低质量报告。需要增加低质量检索兜底。

### 涉及文件

- `deep_research/app/mult_agents/nodes.py`
- `deep_research/app/mult_agents/state.py`
- `deep_research/app/mult_agents/graph.py`

### 实现思路

1. 在 web/local 检索后统计：
   - raw_count
   - kept_count
   - dropped_count
   - avg_relevance_score
2. 如果 kept_count = 0 或 avg_relevance_score 低于阈值：
   - 设置 `retrieval_quality = "low"`。
   - 设置 `needs_more_research = True`。
   - 写入 `missing_gaps`。
3. analyze 节点结合检索质量决定是否 reflect。
4. reflect 节点生成更具体的补充查询词。

### 验收标准

- Web 检索结果全部低相关时不会直接生成强结论。
- 系统能进入 reflect 生成补充检索计划。
- 超过 max_iterations 后仍能输出带风险提示的报告。

## 11. 改造任务九：deep_research 反思循环防护

### 目标

当前 `needs_more_research` 主要依赖 LLM 判断，虽然有 `max_iterations`，但仍需要额外保护，避免无意义补检索。

### 涉及文件

- `deep_research/app/mult_agents/graph.py`
- `deep_research/app/mult_agents/nodes.py`
- `deep_research/app/mult_agents/state.py`

### 实现思路

1. 保留 `max_iterations` 作为硬上限。
2. 增加 supplementary_queries 去重。
3. 如果连续两轮补检索没有新增有效 evidence，则强制进入 write。
4. 如果新证据数量低于阈值，则降低继续反思的优先级。
5. 如果 missing_gaps 与上一轮完全相同，则停止反思。

### 验收标准

- 不会无限循环。
- 无新增证据时能停止补检索。
- 最终报告中能说明证据不足和风险。

## 12. 改造任务十：用户身份与多租户隔离

### 目标

当前部分接口中 `user_id / tenant_id / thread_id` 来自请求体。生产环境中不能信任前端传来的身份字段。

### 涉及文件

- `cloud_agent/app/router/chat.py`
- `cloud_agent/app/service/chat_service.py`
- `deep_research/app/backend/router/research_router.py`
- `deep_research/app/backend/service/workflow_service.py`

### 实现思路

1. 增加认证依赖，例如 `get_current_user()`。
2. 从 token/session 中获取可信 `user_id`。
3. 请求体中的 user_id 仅用于本地调试，生产模式忽略。
4. Redis key、Milvus filter、SQL 查询、checkpointer thread_id 都带上可信 user_id / tenant_id。
5. 日志中不直接输出明文 user_id，可输出 hash。

### 验收标准

- 前端传 `user_id=user_1002` 不会影响后端真实用户身份。
- 工具调用中的 user_id 始终来自认证上下文。
- 多用户并发请求不会串 session、memory 或 cache。

## 13. 改造任务十一：结构化日志与 request_id

### 目标

当前系统使用 `print` 和普通 `logger.info` 较多，缺少贯穿链路的 request_id。需要先建立最小可用的结构化日志能力。

### 涉及文件

- `cloud_agent/app/app_main.py`
- `cloud_agent/app/router/chat.py`
- `cloud_agent/app/service/chat_service.py`
- `deep_research/app/app_main.py`
- `deep_research/app/backend/service/workflow_service.py`
- 各 Agent 节点文件

### 实现思路

1. API 入口生成 `request_id`。
2. 将 `request_id` 放入 `AgentState.metadata` 或 runtime config。
3. 所有日志包含：
   - request_id
   - user_id_hash
   - session_id / thread_id
   - agent_name / node_name
   - event_type
   - latency_ms
4. 用 logger 替换核心 print。
5. 日志输出 JSON 格式，便于后续接入 ELK / Loki。

### 验收标准

- 一次请求能通过 request_id 串起 API、Agent、Tool、Memory 日志。
- 日志不泄露敏感明文 user_id。
- Agent 路由、工具调用、RAG 检索、降级事件都可追踪。

## 14. 改造任务十二：指标监控

### 目标

增加最小可用指标，方便量化项目效果，也方便面试时说明数据来源。

### 推荐指标

- 请求指标：
  - `request_total`
  - `request_success_total`
  - `request_error_total`
  - `request_latency_ms`
- 路由指标：
  - `route_total{agent=...}`
  - `route_fallback_total`
- LLM 指标：
  - `llm_call_total`
  - `llm_latency_ms`
  - `llm_prompt_tokens`
  - `llm_completion_tokens`
- 工具指标：
  - `tool_call_total{tool=...}`
  - `tool_error_total{tool=...}`
  - `tool_latency_ms{tool=...}`
- RAG 指标：
  - `rag_query_total`
  - `rag_retrieved_count`
  - `rag_kept_count`
- Memory 指标：
  - `memory_short_available`
  - `memory_long_available`
  - `memory_retrieve_count`
  - `memory_degraded_total`
- Cache 指标：
  - `semantic_cache_hit_total`
  - `semantic_cache_miss_total`

### 验收标准

- 能统计缓存命中率。
- 能统计每个 Agent 路由分布。
- 能统计每个工具调用耗时和失败率。
- 能统计 Redis / Milvus 降级次数。

## 15. 建议开发顺序

### 第一阶段：功能完整性

1. 新增 `support_agent`。
2. 新增 `fallback_agent`。
3. 优化 Orchestrator 路由规则。
4. 编写路由测试集。

### 第二阶段：安全与稳定性

1. 接入 request_id。
2. 增强 UserId 隔离。
3. 增加工具调用审计。
4. 增加 Redis / Milvus 降级日志。
5. 增加 FinOps 事实校验。

### 第三阶段：性能与工程化

1. 实现 MCPToolRegistry。
2. 缓存工具列表。
3. 增加工具调用 timeout。
4. 接入中途偏好提取。
5. 优化 deep_research 反思循环。

### 第四阶段：可观测性与生产化

1. 结构化 JSON 日志。
2. Prometheus 指标。
3. Grafana 看板。
4. OpenTelemetry Trace。
5. 成本统计和缓存收益统计。

## 16. 面试表达总结

可以这样概括本轮工程化改造：

> 当前项目已经实现了多 Agent 编排、RAG、语义缓存、记忆和 MCP 工具调用。后续工程化改造重点不是继续堆功能，而是提升系统的可控性和生产可用性。我会先补齐 support/fallback 等业务边界，再优化 Orchestrator 的复合意图路由；然后把 MCP Client 从每次请求创建改成应用级生命周期管理，减少高并发下的重复初始化和资源风险；同时加强 user_id 认证上下文、工具调用审计、FinOps 事实校验和 deep_research 的反思循环防护。最后补齐 request_id、结构化日志、指标和 trace，让系统可以量化路由准确率、缓存命中率、Token 成本、工具失败率和降级次数。
